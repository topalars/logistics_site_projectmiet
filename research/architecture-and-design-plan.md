# Архитектурный план и Дизайн-система WMS (Ozon-like)

## 1. Введение
Этот документ объединяет функциональные требования из PRD и лучшие практики современного UI/UX дизайна для создания высокопроизводительной системы управления складом. План ориентирован на создание масштабируемого, быстрого и эргономичного продукта.

---

## 2. Дизайн-система и UX Принципы

### 2.1. Философия дизайна: "Focus on Data & Action"
Интерфейс не должен отвлекать. Главная цель — скорость принятия решений и выполнения физических действий.
*   **Принцип "Один экран — одна задача":** Избегать перегруженности. На экране приемки — только данные о приемке.
*   **Визуальная иерархия:** Размер шрифта и цвет определяют важность. Штрихкод и количество всегда видны сразу.
*   **Снижение когнитивной нагрузки:** Использование иконок для статусов, цветовое кодирование (Green/Red/Amber), предзаполнение полей.

### 2.2. Цветовая палитра (Semantic Colors)
Используем палитру, безопасную для дальтоников и контрастную для складского освещения.
*   **Primary (Бренд):** Глубокий синий (`#005BFF`) — для основных действий и навигации.
*   **Success:** Изумрудный (`#10B981`) — выполнено, в наличии, отгружено.
*   **Warning:** Янтарный (`#F59E0B`) — ожидание, низкий остаток, требует проверки.
*   **Error:** Красный (`#EF4444`) — брак, недостача, ошибка сканирования.
*   **Background:** Светло-серый (`#F3F4F6`) — для фона приложения, белый (`#FFFFFF`) — для карточек.
*   **Text:** Темно-серый (`#111827`) — основной текст, светло-серый (`#6B7280`) — второстепенный.

### 2.3. Типографика
*   **Шрифт:** `Inter` или `Roboto` (гротески с высокой читаемостью).
*   **Размеры:**
    *   Базовый текст: 16px (не менее, для читаемости с расстояния).
    *   Заголовки карточек: 18px Bold.
    *   Цифровые данные (количество, артикулы): 20px+ Mono space (для выравнивания).
*   **Line-height:** 1.5 для основного текста, 1.2 для заголовков.

### 2.4. Компоненты (Atomic Design)
Библиотека компонентов должна быть строго типизирована и переиспользуема.
1.  **Atoms:** Кнопки (Large/Medium/Small), Инпуты (с поддержкой сканеров штрихкодов), Бейджи статусов, Иконки.
2.  **Molecules:** Карточка товара, Строка таблицы с действиями, Форма поиска, Фильтры.
3.  **Organisms:** Таблица заказов, Карта склада (Grid view), Дашборд статистики, Модальное окно подтверждения.
4.  **Templates:** Layout страницы (Sidebar + Header + Content), Страница задачи (Task View).

### 2.5. Специфика складского UX
*   **Touch Targets:** Минимальный размер кнопок 44x44px (для работы в перчатках).
*   **Режим ТСД (Terminal Mode):** Упрощенный интерфейс для мобильных устройств: крупные кнопки "Да/Нет", фокус на поле сканирования по умолчанию.
*   **Звуковая обратная связь:** Визуальные дубли звуковых сигналов сканера (зеленая вспышка при успехе, красная при ошибке).
*   **Темная тема:** Обязательна для работы в ночные смены или в зонах с плохим освещением.

## 3. Техническая Архитектура (Frontend)

### 3.1. Стек технологий
*   **Фреймворк:** React 18+ (или Vue 3) — для компонентного подхода и реактивности.
*   **Язык:** TypeScript — строгая типизация критична для логистических данных.
*   **Сборщик:** Vite — для мгновенного старта и быстрой сборки.
*   **Стилизация:** Tailwind CSS — для скорости разработки и консистентности дизайн-системы.
*   **Управление состоянием:** Zustand или TanStack Query — для серверного состояния и кэширования.
*   **Роутинг:** React Router v6+ с поддержкой вложенных маршрутов.

### 3.2. Структура проекта (Feature-Sliced Design подход)
Разделение по бизнес-фичам, а не по типу файлов.

```
src/
├── app/                  # Глобальные настройки, провайдеры, роутер
├── pages/                # Страницы (сборки из фич)
│   ├── dashboard/
│   ├── inbound/          # Приемка
│   ├── storage/          # Хранение
│   ├── outbound/         # Отгрузка
│   └── settings/
├── widgets/              # Крупные блоки (Header, Sidebar, MapWidget)
├── features/             # Бизнес-логика (ScanBarcode, CreateOrder, MoveItem)
├── entities/             # Бизнес-сущности (Product, Order, Cell, User)
├── shared/               # Переиспользуемый код (UI Kit, hooks, utils, api)
│   ├── ui/               # Дизайн-система (Button, Input, Modal)
│   ├── lib/              # Утилиты
│   └── assets/           # Иконки, шрифты
└── main.tsx
```

### 3.3. Производительность и Оптимизация
*   **Virtual Scrolling:** Для таблиц с тысячами строк (списки товаров, заказы).
*   **Code Splitting:** Ленивая загрузка страниц (роутинг по частям).
*   **Offline First (PWA):** Service Workers для работы при временной потере связи на складе (кэширование операций, очередь синхронизации).
*   **Web Workers:** Вынос тяжелых вычислений (расчет маршрутов, фильтрация больших списков) в отдельные потоки.

## 4. Техническая Архитектура (Backend & Data)

### 4.1. Архитектурный стиль
*   **Modular Monolith (на старте):** Единая кодовая база, но с четким разделением на модули (Inbound, Outbound, Inventory). Легко масштабируется в микросервисы позже.
*   **Event-Driven:** Внутренняя коммуникация через события (например, `OrderCreated` -> `ReservationStarted` -> `TaskGenerated`).

### 4.2. База данных
*   **PostgreSQL:** Основное хранилище (транзакционность, целостность).
*   **Redis:** Кэширование сессий, горячих данных (остатки в реальном времени), очередей задач.
*   **TimescaleDB (опционально):** Для хранения истории перемещений и телеметрии.

### 4.3. API
*   **REST / GraphQL:** REST для простых CRUD, GraphQL для сложных выборок данных на дашборд.
*   **WebSocket:** Для пуш-уведомлений (новые задания, изменения статусов в реальном времени).
*   **Версионирование:** Строгое версионирование API (`/api/v1/...`).

## 5. План реализации (Roadmap)

### Фаза 1: Фундамент и Дизайн-система (Неделя 1-2)
1.  **Настройка окружения:** Init репозитория, настройка TS, ESLint, Prettier, Husky.
2.  **Разработка UI Kit:** Создание базовых компонентов (Button, Input, Table, Card, Badge) согласно дизайн-коду.
3.  **Layout:** Создание оболочки приложения (Sidebar, Header, Mobile Nav).
4.  **Роутинг:** Настройка структуры страниц.
5.  **Mock Data:** Генерация реалистичных тестовых данных (товары, ячейки, заказы).

### Фаза 2: Ядро логики - Приемка и Хранение (Неделя 3-4)
1.  **Модуль "Товары":** Каталог, карточка товара, штрихкоды.
2.  **Модуль "Приемка":**
    *   Создание накладной.
    *   Интерфейс сканирования (эмуляция ввода сканера).
    *   Визуализация процесса приемки.
3.  **Модуль "Хранение":**
    *   Визуальная карта склада (Grid/SVG).
    *   Логика размещения (алгоритм подбора ячейки).
    *   Перемещение товаров между ячейками.

### Фаза 3: Отгрузка и Заказы (Неделя 5-6)
1.  **Модуль "Заказы":** Список, фильтры, приоритеты.
2.  **Модуль "Комплектация" (Picking):**
    *   Генерация заданий на подбор.
    *   Маршрутизация по складу (оптимизация пути).
    *   Интерфейс подбора (сканирование -> подтверждение).
3.  **Упаковка и Отгрузка:** Фиксация веса, печать этикеток (макет).

### Фаза 4: Аналитика и Полировка (Неделя 7-8)
1.  **Дашборд:** Графики, KPI, виджеты активности.
2.  **Адаптивность:** Тестирование и правка под планшеты/ТСД.
3.  **Темная тема:** Реализация переключателя и стилей.
4.  **Производительность:** Профилирование, оптимизация рендеринга.
5.  **Документация:** Readme, инструкция пользователя.

## 6. Критерии качества (Definition of Done)

1.  **Дизайн:** Полное соответствие утвержденной дизайн-системе. Все интерактивные элементы имеют состояния (hover, active, disabled, focus).
2.  **Код:** Отсутствие ошибок TypeScript. Покрытие тестами критических путей (сканирование, расчет остатков) > 80%.
3.  **Производительность:** First Contentful Paint < 1.5s. Интерфейс реагирует на ввод < 100ms.
4.  **UX:** Сценарий "Приемка товара" выполняется минимальным количеством кликов. Ошибки сканирования обрабатываются понятно.
5.  **Адаптивность:** Корректное отображение на Desktop (1920x1080), Tablet (1024x768) и Mobile (360x640).

## 7. Риски и Митигация

| Риск | Влияние | Митигация |
|------|---------|-----------|
| Сложность реализации карты склада | Высокое | Использовать готовую библиотеку для гридов или SVG, не писать с нуля движок рендеринга. |
| Проблемы с работой сканеров в вебе | Среднее | Использовать стандартный ввод (клавиатурный режим сканеров) + поддержка Camera API для мобильных. |
| Перегруженность интерфейса | Высокое | Строгий аудит компонентов с дизайнером. Приоритет простоте над "красивостью". |
| Производительность на слабых ТСД | Среднее | Упрощенный режим интерфейса для мобильных, агрессивное кэширование. |
